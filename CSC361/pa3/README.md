# Assignment 3

This Python program is a Traceroute Traffic Analyzer. It reads a raw binary PCAP/CAP network packet capture file to reconstruct and analyze the path data packets took across the Internet to reach a destination host.
Crucially, it is cross-platform aware: it can handle trace files generated by Windows traceroute (which uses ICMP Echo Requests) as well as Linux/Unix traceroute (which uses UDP probe packets with high-numbered destination ports).

## Core Architecture and Header Parsing
The script acts as a manual decoder for binary packet headers without using outside capture frameworks like Scapy. It uses Python's struct module to unpack binary data buffers into readable properties across several specialized classes:
- GlobalHeader: Reads the initial 24 bytes of the PCAP file to parse system metadata like file format magic numbers, version details, and network link type.
- PacketHeader: Unpacks individual packet wrappers (16 bytes each) to capture capture lengths and raw microsecond timestamps.
- IPHeader: Decodes the IPv4 layer. It calculates the dynamic Internet Header Length (IHL), parses total packet length, reads identification/fragmentation flags, and translates binary fields into standard dot-decimal IP strings (e.g., 192.168.1.1).
- UDPHeader / ICMPHeader: Conditional transport layers. If ICMP is active, it extracts message types (like 8 for Echo Request or 11 for Time Exceeded). If an ICMP error packet is captured, it deep-dives into the payload to extract the original packet's inner header metadata.

## Step-by-Step Execution Flow
### 1. Sequential File Ingestion
The program reads the input PCAP file sequentially packet-by-packet. It skips the 14-byte physical Ethernet frame layer and reads the underlying IP protocol. It filters out everything except ICMP (Protocol 1) and UDP (Protocol 17) packets, building chronological arrays of outbound probes and inbound replies.

### 2. OS Variant Determination & Packet Matching
Because different operating systems perform traceroutes differently, the script splits its logic:
- Windows Mode: Activated if outbound packets contain ICMP Echo Requests (Type 8). It pairs them with inbound replies by matching their structural ICMP Sequence Numbers.
- Linux Mode: Activated when outbound traffic contains high-port UDP probes (ports 33434 to 33529). It pairs these outbound requests with inbound ICMP Time Exceeded errors by matching the outbound UDP Source Port with the port embedded inside the returned ICMP error frame.

### 3. IP Fragmentation Tracking
The script groups outbound packets using their IP Identification fields.
- If multiple packets share the same ID, it flags the datagram as fragmented.
- It counts the total fragments generated per transmission and computes the bitwise byte-offset of the final fragment slice using the fragment_offset properties.

### 4. Round-Trip Time (RTT) Metrics
For every successfully matched probe and reply pair, the program computes the precise network latency. It converts the packet's raw microsecond timestamps into milliseconds relative to the start of the capture, finding the exact time delta ($t_{\text{reply}} - t_{\text{probe}}$)

## Program Output Summary
At completion, the script prints out a diagnostic network performance report containing:
- Network Topology Summary: The explicit IP address of the scanning origin node, the final target node, and an ordered list of every intermediate router hop discovered along the network pathway.
- Protocol Audit: A breakdown list of every transport protocol encountered inside the capture file.
- Fragmentation Report: Detailed counts of packet slicing, stating how many fragments were generated from larger original datagrams along with their absolute tail-end alignment offsets.
- Statistical Performance Breakdown: For each network router hop, it outputs the average Round-Trip Time (RTT) alongside its statistical standard deviation ($\sigma$) in milliseconds, illustrating path latency consistency.
